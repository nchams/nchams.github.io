Automating Daily Reports from IoT Care Sensors

From raw cloud sensor streams to a report care teams actually read every morning.

Role
Data engineering & reporting automation, end to end
Domain
IoT health-monitoring sensors deployed across senior-living communities
Stack
Python, AWS Timestream (boto3), pandas/NumPy, Plotly, self-contained HTML reporting

The problem

A health-technology company deploys IoT sensors in senior-living communities to detect incontinence events, so caregivers respond in minutes instead of hours. The sensors stream constantly to a cloud time-series database. The data was all there. The visibility wasn't.

Care directors and operations staff needed to know, every morning: what happened overnight, room by room? How quickly did staff respond? And which devices need attention? Getting those answers meant someone technical digging through dashboards and raw queries, which doesn't scale across multiple operators, communities, and time zones.

The brief: turn the raw streams into an automated daily report that a non-technical care team can open and act on, with no logins, no dashboards, and no training.

The pipeline

We built a report engine in Python that runs against AWS Timestream, the platform's time-series database, and handles the messy realities of a real device fleet:

  • Per-community time zones: each community's "day" runs 9:00 AM to 8:59 AM local time. The engine resolves every community's timezone from org metadata and converts each reporting window to UTC correctly, including daylight-saving transitions.
  • Per-device thresholds: every sensor has its own calibrated alert threshold pulled from device metadata, with a safe fallback when one is missing, so charts and event logic reflect what each device actually does.
  • Efficient queries at fleet scale: full-day sensor traces are downsampled server-side into 10-second bins inside the SQL query, so the report stays fast and light even with a full day of high-frequency data per room.
  • Resilient cloud plumbing: endpoint discovery, retry policies, and paginated result parsing so a transient AWS hiccup doesn't kill the morning report.

What the report contains

One HTML file per operator, with a dropdown to switch between communities. Each community section gives staff the full picture:

  • Summary KPIs for the reporting window;
  • Incontinence events by room, so patterns and high-need residents stand out;
  • Event response times, the accountability metric: how long between the sensor alert and the caregiver's response;
  • Operational issues: rooms offline (detected two ways, from uptime rollups and from the latest device service status), plus suspect pads and modules flagged from tagged annotations;
  • Full-window interactive sensor traces for every room, with each device's own threshold drawn on the chart and detected events shaded directly on the trace, so anyone can see exactly what the sensor saw.

The whole report is a single self-contained HTML file: logo embedded, styling inlined, charts interactive. It opens in any browser from an email attachment. No server, no accounts, no IT involvement at the care community.

Built to scale across operators

Everything specific to a customer lives in a small configuration block: the operator's name, their list of communities, and the reporting window. The same engine produces branded daily reports for multiple senior-living operators, from a single-community deployment to operators with five or more communities in different time zones. Onboarding a new operator is a config change, not a new codebase.

What this project demonstrates

  • Cloud time-series engineering: efficient, resilient querying of AWS Timestream across a live device fleet, with server-side downsampling to keep reports fast.
  • Reporting people actually use: a self-contained, interactive report designed for non-technical care staff, not a dashboard that assumes an analyst.
  • Operational awareness built in: device health and data quality (offline rooms, suspect hardware) reported alongside the care metrics, because a silent sensor is a care risk.
  • Multi-tenant thinking: per-community time zones, per-device thresholds, and config-driven branding so one engine serves many customers.
