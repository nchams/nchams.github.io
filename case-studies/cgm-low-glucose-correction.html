Reducing Low-Glucose Error in a CGM Algorithm

A guardrailed machine-learning correction layer for continuous glucose monitoring.

Role
Algorithm research & development: design, modeling, evaluation, and production integration
Domain
Continuous glucose monitoring (CGM), medical-device signal processing
Stack
Python, XGBoost, pandas/NumPy, a graph-based real-time signal pipeline

The problem

Continuous glucose monitors are most safety-critical exactly where they tend to be least accurate: the hypoglycemic range (low blood glucose).

A missed or mis-stated low reading is the failure mode that matters most to a person with diabetes, yet it's where the sensor's relative error is highest. Low-glucose accuracy lagged well behind the device's overall accuracy.

The goal was to reduce low-glucose error without destabilizing normal-range behavior or introducing unsafe corrections: a hard constraint in a regulated medical device, where "improve accuracy on average" is not acceptable if it makes any individual scenario worse.

The approach

Rather than predict glucose directly, we built a model that predicts the error between the sensor's estimate and a reference measurement:

predicted_error = reference_glucose − current_sensor_value

A single gradient-boosted model (XGBoost) learns this error from a feature vector describing the live signal: current value, filtered raw signal, sensor age, and multi-window statistics (rate-of-change, acceleration, and their normalized forms across several lookback windows).

The model's output never touches the live reading directly. It passes through a deterministic guardrail layer that encodes the safety contract:

  • corrections are only allowed when the current reading is in the low-glucose region;
  • the raw prediction is clipped, then shrunk by a conservative factor;
  • the maximum correction magnitude is hard-bounded;
  • if anything fails at runtime, the system passes the original reading through unchanged (fail-open).

This separation, a learned component wrapped in auditable, deterministic rules, is what makes a machine-learning model defensible inside a safety-critical device.

Evaluation discipline

Everything was measured with leave-one-study-out cross-validation: the model is repeatedly trained on all clinical datasets but one, then evaluated on the held-out data. This prevents optimistic results from a model that has effectively seen its test subjects, and it surfaces per-dataset regressions that a pooled average would hide.

The acceptance bar was deliberately strict: an improvement only counts if it lowers low-glucose error and no individual dataset gets worse on either low-range or overall accuracy.

Results

Evaluated across multiple clinical datasets using the field-standard accuracy metric (MARD, mean absolute relative difference):

  • A meaningful reduction in low-glucose error, the range that matters most for safety.
  • An even larger improvement against high-precision laboratory reference measurements.
  • Normal-range accuracy preserved, with corrections concentrated almost entirely in the low-glucose region.
  • No dataset regressed on low-range or overall accuracy. The safety bar was met.

Going further: aligning the objective with the metric

A model is only as good as what it's optimizing. We noticed a mismatch worth chasing: the accuracy metric is a relative error measure, but the model minimizes squared absolute error, which structurally under-weights low-glucose points, where the same absolute miss is a much larger relative error.

We designed and tested a metric-aware sample-weighting scheme that turns the model's loss function into a surrogate for relative error, concentrating the model's capacity on the lowest-glucose readings continuously rather than via a hard cutoff. We swept the parameters through the same cross-validation harness, holding the guardrails fixed so the comparison isolated the training objective.

The experiment produced a nuanced, honest result rather than a headline:

  • Pure relative weighting didn't improve low-glucose error, but consistently improved accuracy in the adjacent low-normal range, a real side benefit.
  • Simply increasing the low-glucose weight pushed error down further, but at the cost of one dataset regressing, failing the safety bar.
  • A hybrid scheme found the safe win: a further simultaneous reduction in both low-glucose and low-normal error, with zero dataset-level regressions.

The change was a single function in the training code. Features, guardrails, the exported model bundle, and the real-time inference path were all untouched, so it carried essentially no integration risk. The experiment also produced a tunable tradeoff curve for the product team to set policy against.

From research to production

The work didn't stop at offline modeling. We built the path to a live device:

  • a frozen model bundle (model + a manifest that serves as the runtime contract: feature names, guardrail settings, versioning) loaded once at startup and reused for every reading, with no retraining in the field;
  • a real-time correction node in the signal-processing graph that reconstructs the exact feature vector from live signal history and applies the identical guardrails used offline;
  • staged rollout modes: disabled, shadow (compute but don't act), and active (downstream alerts and predictions consume the corrected value), so the layer can be validated in production without affecting patients before it's trusted;
  • fail-open behavior throughout: a missing bundle, malformed features, or an inference error all pass the original reading through unchanged.

What this project demonstrates

  • Safety-critical ML design: pairing a learned component with a deterministic, auditable guardrail layer so the system is defensible in a regulated context.
  • Evaluation rigor: leave-one-study-out validation and a per-dataset no-regression bar, not just average-case wins.
  • Objective-metric alignment: diagnosing that the loss function was structurally misaligned with the accuracy metric, then designing and testing a principled fix.
  • End-to-end ownership: from feature engineering and modeling through to a production-compatible, fail-open real-time inference path with a staged rollout strategy.
